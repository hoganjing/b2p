# 目录 / 路径 / 产物 / 日志 规范（单一事实来源）

本文件是 `book-to-podcast` 技能在「文件落在哪、技能目录能不能写」上的**唯一硬性约束**。
SKILL.md 讲做什么，workflow.md 讲怎么做；本文件讲**产物与目录边界**——任何调用者
（人类、主 agent、子代理、后台进程）都必须遵守，不靠默契、不靠自由发挥。

> 设计原则：**技能目录 `skills/b2p/` 是只读的共享代码资产；所有项目产物只许落在调用方
> 提供的工作目录（项目目录）里。** 脚本默认用相对路径 `scripts/xxx.py` 运行，输入/输出
> 都相对「当前工作目录」，因此「当前工作目录 = 项目目录」是铁律。

---

## 0. 红线（违反即错）

- **禁止向技能目录写入任何项目产物。** 不得把 `.md`、`.tts.txt`、`.mp3`、`.log`、临时
  草稿、提取脚本副本等复制/生成进 `skills/b2p/`（含其下的 `scripts/`、`scripts/audio/`、
  `references/`）。技能目录只允许：运行 `scripts/` 里的代码、读取 `references/` 配置、
  以及 `scripts/anchor/` 里的参考音频资产。
- **技能目录里的文件只增不减、且只由「维护技能」的动作改动。** 跑一次书的处理，不应
  让 `skills/b2p/` 多出任何文件，也不应删/改其中任何文件（除非你在显式地升级技能本身）。
- 若某脚本把输出硬钉在技能目录（见 §3 的已知坑），**不要靠复制绕过**——用该脚本提供的
  重定向参数（如 `--out-dir`），或把输入也放进项目目录、跑完即清，但绝不让产物留在技能目录。

---

## 1. 工作目录（你在哪工作）

- **当前工作目录（CWD）必须是项目目录**，例如 `E:\poca\A Brain for Innovation\podcast\`。
- 运行任何 `scripts/xxx.py` 时，从项目目录调用，并让脚本的「相对路径」解析到项目目录：
  ```bash
  cd "<项目目录>"
  python "<技能目录>/scripts/make_tts.py" .
  python "<技能目录>/scripts/audit.py" . --book <slug> --briefs "briefs"   # briefs/ 在你的项目目录内
  ```
- 脚本内部用 `os.path.dirname(__file__)` 定位**自己的代码/配置**（apis.yaml、tts_engine 包、
  anchor 资产）——这部分本来就指向技能目录，是只读引用，**无需也不应改动**。
- 脚本内部用「当前工作目录」或显式参数定位**输入/输出**（`.tts.txt`、`.mp3`、日志）——
  这部分必须落在项目目录。两件事不要混：代码在技能目录（只读），数据在项目目录（读写）。

---

## 2. 项目目录里建哪些文件夹、生成哪些文件

一个典型项目目录的结构（**全部由你在项目目录内创建，技能目录不替你建**）：

```
<项目目录>/
├── source/                 # [可选] 抽取出的每章原文 .txt（阶段〇/阶段一的输入）
├── *.md                   # 每章口播稿（人/主播看，含括号标签与主播延伸）
├── *.tts.txt              # 每章干净稿（喂 TTS，由 make_tts.py 从 .md 生成）
├── audio/                 # TTS 生成的 .mp3（或 .wav 视 provider）
├── _backup_full_v1/       # [可选] 长章压缩前的完整版备份
└── (日志见 §4)
```

- **`.md` 与 `.tts.txt` 成对**：`第N章·译名.md` + `第N章·译名.tts.txt`，同名不同后缀。
- **`.tts.txt` 由 `make_tts.py` 生成**，不要手改；改稿改 `.md` 后重跑 `make_tts.py`。
- **音频落在 `audio/`**（相对于项目目录的 `audio/`），不要落在别处，也不要落在技能目录。
- 临时脚本/探针（如 PDF 渲染、提取脚本）也放在项目目录（如 `_probe/`），跑完即清，**绝不**
  放进技能目录。

## 3. 已知坑：脚本把路径钉死在自身目录时怎么办

- `tts_run.py` 内部 `AUDIO_DIR = os.path.join(HERE, "audio")`，且用
  `glob(HERE/*.tts.txt)` 找输入——即**默认把输入/输出都钉在 `scripts/` 目录**。
- **正确做法（任选其一，按推荐度排序）：**
  1. **优先用 `--out-dir`**：`python tts_run.py --api edge_tts --out-dir "<项目目录>/audio"`
     （把音频重定向到项目目录；输入 `.tts.txt` 仍需在脚本能找到处，见下）。
  2. 若输入也必须隔离：在**项目目录内**建一个临时子目录，把 `.tts.txt` 复制进去、**从那个
     项目子目录**运行 `tts_run.py`（`HERE` 即该子目录），音频落到该子目录的 `audio/`，**跑完
     把音频移回项目 `audio/`、删掉临时子目录**。**全程不碰技能目录。**
  3. 不要选「复制进 `skills/b2p/scripts/` 再跑」——这直接违反 §0 红线。
- `make_tts.py` / `audit.py` 用「当前工作目录」定位输入/输出（`argparse` 的 `dir` 默认
  `os.getcwd()` 或脚本所在目录），从项目目录运行即可，不会污染技能目录。

## 4. 日志（要不要、在哪）

- **TTS 运行建议带日志**，便于监控长连接/卡死（见 workflow.md §2.1–2.3 的看门狗）。
- **日志落在项目目录的 `audio/` 下**，文件名 `gen.gN.log`（N 为批次号），例如
  `audio/gen.g1.log`。多组并行用不同 N 避免交错。
- 看门狗 `watchdog.py` 默认也在项目 `podcast/` 运行、日志默认同级 `audio/`。
- **日志严禁写在技能目录。**
- 日志为**可选但推荐**；纯本地一次性小批量（如 edge_tts 几章）无日志也可，但长批/后台批
  必须带日志 + 看门狗，否则卡死不可观测。
- 日志是项目产物，处理完可保留也可清，随你；但同样**只许在项目目录内**。

---

## 5. 收尾自检（每次交付前必做，双向验证）

1. **项目目录齐全**：`*.md`、`*.tts.txt`、`audio/*.mp3` 数量一致（每章一对稿 + 一个音频）。
2. **技能目录零污染**：`skills/b2p/` 下除原有代码/配置/anchor 资产外，**没有任何本次新增的
   `.md`/`.tts.txt`/`.mp3`/`.log`/临时文件**。`scripts/audio/` 应保持空或不存在。
3. **audit 门禁通过**：`audit.py . --book <slug>` → `noise=0` 且 `strayEN=0`。
4. 音频健康：每章 `字节/字` 比值 ≥ 基线（参考 workflow.md §2.2，约 2500–2900 B/字），
   无截断。

> 第 2 条是本次新增的关键约束：**不仅要验证「交付物在不在项目目录」，还要验证
> 「共享技能目录干不干净」。** 只朝一个方向看会漏掉污染。
